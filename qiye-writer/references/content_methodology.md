## 1. 选题方法论

### 选题交集模型

一个好选题 = 你正在折腾的东西 + 同行可能遇到的问题 + 当下的时间节点

光是自己折腾了，可能太小众。光是蹭热点，可能没深度。光是解决了一个问题，可能太简单不值得写。三者结合才能找到最有价值的选题。

### 选题来源

**日常开发**：你今天在配置环境、写代码、调试的时候遇到了什么问题？这个问题你是怎么解决的？这就是最好的选题。

**GitHub Trending**：每天刷一下 GitHub Trending，看看有什么新项目。特别是跟你技术栈相关的。你不需要每个都写，但看到一个有意思的，可以花半小时试一下，有感受就写。

**技术社区**：V2EX、掘金、知乎、Reddit（r/programming、r/ClaudeAI 等），看看大家在讨论什么。评论区里经常藏着好的选题。

**工具更新日志**：Claude Code 更新了什么？WSL2 有什么新功能？uv 又发了什么版本？版本更新日志里经常藏着值得深挖的点。

**读者反馈**：你之前的文章评论区，有没有人追问某个细节？有没有人说「能不能写一篇关于 xxx 的」？这些就是最直接的需求。

**踩坑经历**：你花了很久解决的一个 bug，大概率别人也会遇到。这种文章天然有搜索流量。

### HKR 质检法

动笔前每个选题必须通过：

- **H (Hands-on)**：有真实动手吗？是亲测有效的还是纸上谈兵？
- **K (Knowledge)**：读者看完能带走什么？能用的配置？能理解的概念？能避开的坑？
- **R (Resonance)**：能戳中程序员的痛点吗？让人「对对对我也遇到过」？

三项至少占两项。纯搬运官方文档没有价值，除非你加了大量自己的实操理解。

### 角色代入法

审视选题时想象一个读者：

- 「一个跟你一样在 Windows 上搞开发的 Java 后端」：他能直接照搬你的方案吗？
- 「一个刚开始学 AI 编程的新手」：他能看懂你的解释吗？会不会跳步太多？
- 「一个在犹豫要不要用这个工具的人」：你的文章能帮他做决定吗？

## 2. 选题分类与过往案例

### 环境搭建 / 踩坑记录

核心是「我帮你趟过这条河了」。读者看这类文章就是为了照着做。

**过往案例：**
- Windows + WSL2 + AI Coding 最佳姿势（置顶文章，最高阅读）
- Claude Code 安装 & 配置（56 阅读，4 点赞）
- Windows + Docker Desktop + WSL2 最佳实践（52 阅读）
- WSL2 运行 Claude Code 卡顿的解决方法
- 使用 nvm 安装管理 Node 环境
- 最强 Python 管理器 uv
- WSL 配置 GitHub SSH

**写法要点：**
- 步骤必须完整，不能跳步。读者最怕的就是「然后按照官方文档配置好就行」
- 每一步都要有可以照搬的命令
- 踩坑点要标出来，最好附上报错信息
- 说明你的环境版本（Windows 版本、WSL2 内核版本、Node 版本等）

### 技术原理 / 深度解析

核心是「这个东西到底怎么运作的」。读者看完要从「知其然」变成「知其所以然」。

**过往案例：**
- Claude Code 七层可编程架构（31 阅读，4 点赞，热度高）
- Claude Code 工作原理 & 记忆体系
- 规范驱动开发 SDD

**写法要点：**
- 用类比把抽象概念翻译成大白话
- 层层剥开，不要一次性丢出全部信息
- 配合架构图描述或流程描述
- 最后一定要落回到「所以这对我日常开发有什么用」

### 工具对比 / 评测

核心是「我帮你试过了，你该怎么选」。读者不想自己每个都试一遍。

**过往案例：**
- AI 编程工作流对比（Spec-Kit vs OpenSpec vs Superpowers）
- IDEA 2026.1 + Qoder 插件初体验

**写法要点：**
- 每个工具都要实际用过，不是看文档写的
- 有明确的对比维度（上手难度、灵活度、文档质量、适用场景）
- 给明确的推荐，不要和稀泥
- 用表格做总结，一目了然

### 工作流 / 方法论

核心是「这套方法我跑通了，你可以直接用」。读者看完手里要有一个可以执行的方案。

**过往案例：**
- AI 编程规范（4 篇系列）
- 个人知识库搭建（Obsidian 系列）

**写法要点：**
- 每一步都要可执行，不能只有理念没有动作
- 坦诚说明学习成本和常见失败点
- 用你自己的实际工作流做示范
- 开头用谦逊铺垫，不要让人觉得你在教人做事

### 实战项目 / 折腾记录

核心是「跟我一起做」。读者看完要有一种「我也跟着经历了一遍」的参与感。

**过往案例：**
- Obsidian + Hermes + LLM Wiki 实践指南
- Claude Code 宠物系统（娱乐属性拉满）

**写法要点：**
- 完整的英雄之旅叙事弧（遇到问题 → 开始折腾 → 踩坑 → 解决 → 总结）
- 有真实的过程细节，不是只展示最终结果
- 有最终的成果展示（截图、效果对比）
- 坦诚说哪些地方做得好、哪些地方可以改进

## 3. 技术文章的节奏

### 节奏优先于内容密度

一篇技术文章，内容再好，如果读起来像说明书，读者也看不下去。节奏是让人一路往下读的推进系统。

好的节奏：围绕主线讲一个问题，偏出去看个案例或背景，一句话拉回来继续推进。

最忌讳的：突然讲了一大段跟主线无关的东西，然后硬拽回来。读者要花脑力「顺逻辑」，心流断掉。

### 扣主线句

每偏出去一段，就要有一句把读者拉回来。不需要长，一句就够：

- 「回到我们的开发环境配置...」
- 「说了这么多背景，核心其实就是...」
- 「这个原理搞清楚了，接下来看实际操作...」

### 信息密度波动

技术文章不能一直高密度，读者会累。也不能一直水，读者会觉得没干货。

理想的节奏：一个技术要点 → 一个案例或类比（缓冲）→ 下一个技术要点 → 一句吐槽或感受（呼吸）→ 核心结论。

### 疑问句的节奏作用

在讲技术原理的时候，用疑问句制造节奏刹车：

> 为什么 WSL2 会卡呢？
> 因为文件系统的跨层访问。

> 听着很复杂对吧？
> 其实核心就一句话。

第一个疑问句让读者停一秒准备接收新信息，第二个是对读者的共鸣和承诺。

## 4. 代码和配置的展示原则

### 代码块不是孤立存在的

每一段代码前后都要有文字说明。

**好的方式：**
> 先把 WSL2 默认版本设成 2，这个很重要，不设的话后面装的发行版可能还是 WSL1。
> ```bash
> wsl --set-default-version 2
> ```
> 跑完之后没有报错就是成功了。

**坏的方式：**
> 执行以下命令：
> ```bash
> wsl --set-default-version 2
> ```
> 然后继续下一步。

差异：前者解释了「为什么」要做这一步，后者只说了「做什么」。前者让读者理解，后者让读者照搬。

### 命令要可复制

命令必须是完整的、可以直接复制粘贴运行的。不要省略参数，不要用 `...` 代替。

如果一条命令很长，要么保持一行（让读者直接复制），要么用 `\` 换行并说明。

### 输出要有解读

如果命令的输出很重要，贴出关键部分，然后解读：

> 跑完之后会看到类似这样的输出：
> ```
> NAME      STATE    VERSION
> * Ubuntu  Running  2
> ```
> 看到 VERSION 是 2 就说明对了。如果是 1，需要手动转换。

### 配置要说明每个字段

配置文件不要贴一大坨然后说「按需修改」。把关键字段标出来解释：

```yaml
# 这里填你的 API Key
api_key: "sk-xxx"

# 模型选择，推荐 claude-sonnet-4-20250514
model: "claude-sonnet-4-20250514"

# 最大 token 数，太小会截断
max_tokens: 4096
```

### 版本号要标明

技术文章最怕的就是读者照着做结果版本不对。开头说明你的环境：

> 本文基于以下环境，其他版本可能有差异：
> - Windows 11 24H2
> - WSL2 Ubuntu 22.04
> - Claude Code 1.0.x
> - Node.js 20.x (via fnm)

### 错误处理要覆盖

如果某个步骤有可能失败，提前说明：

> 如果这一步报权限错误，试试在前面加 `sudo`。
> 如果还是不行，检查一下你的用户是不是在 `docker` 组里。

不要让读者遇到报错之后再翻评论区找答案。
